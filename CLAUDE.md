# CLAUDE.md

이 파일은 Claude Code (claude.ai/code)가 이 저장소에서 작업할 때 따라야 할 지침을 제공합니다.

## 공통 코딩 규칙

### 핵심 원칙

- **YAGNI (You Aren't Gonna Need It)**: 지금 당장 사용하지 않는 함수나 파일은 만들지 않기
- **KISS (Keep It Simple, Stupid)**: 단순하고 직관적인 코드 작성
- **DRY (Don't Repeat Yourself)**: 코드 중복 금지, 공통 로직은 분리하여 재사용
- **오버엔지니어링 금지**: 필요한 최소 구성만 유지

### 성능 규칙

- **2중 for문(중첩 루프) 금지**
  - Map/Set/Dictionary 인덱싱, 배치 조회, 그룹핑으로 대체
- **N+1 쿼리 금지** (ORM 사용 시)
  - 루프 내 지연 로딩 접근 금지
  - 단일 쿼리 또는 ID 수집 후 IN 조회 사용

### 코드 품질 규칙

- **DTO/VO 사용**: API 응답에 엔티티/모델 직접 노출 금지
- **단일 책임 원칙**: 각 클래스/함수는 하나의 책임만
- **레이어 분리**: Controller/Router → Service → Repository/DAO

### 예외 처리 규칙

- **예외 처리 필수**
  - 외부 API 호출, 파일 I/O, DB 트랜잭션 등 예외 발생 가능한 모든 코드
  - 예외를 무시하지 말고 로깅 또는 적절한 응답 반환
- **유효성 검증 필수**
  - 입력 계층에서 검증 (Controller/Router)
  - 비즈니스 로직 검증 (Service)
  - null/undefined 체크

### Git 커밋 메시지 규칙

**형식:** `[태그] [기능이름] 커밋 설명`

**태그 종류:**
- **[FEA]**: 새로운 기능 추가
- **[REF]**: 코드 리팩토링
- **[BUG]**: 버그 수정
- **[CNF]**: 설정 파일 변경 (application.yml, .env 등)
- **[STYLE]**: 코드 포맷팅 (로직 변경 없음)
- **[DOCS]**: 문서 수정

**주의사항:**
- 커밋 메시지는 한글로 작성
- 한 커밋에는 하나의 논리적 변경사항만 포함
- **Co-Authored-By 사용 금지**


## 핵심 아키텍처

**핵심: 자연스러움 + 정량적 성과**

AI가 작성한 것처럼 보이지 않고, 사람이 자연스럽게 작성한 것처럼 보이면서도 HR에게 임팩트를 주는 표현이 필요합니다.

#### 과장 금지 원칙

**핵심: 사실만 간결하게, 과장하지 않기**

이력서는 자신을 돋보이게 하는 곳이 아니라, **실제로 한 일을 정확히 전달**하는 문서입니다.

- ❌ 중복되고 상세한 설명을 여러 줄로 나열
- ❌ 같은 내용을 다른 표현으로 반복
- ❌ 너무 세세한 프로세스 설명
- ✅ 핵심만 1줄로 간결하게
- ✅ 중복 제거, 통합

**핵심**: AI 티 나는 수식어("효과적으로", "성공적으로", "긴밀한", "지속적인")는 제거하되, **협업, 문제 해결 과정, 주체성은 반드시 유지**해야 합니다.

#### 금지 표현 패턴
- ❌ "~을 통해", "~을 활용한", "~을 위한", "~함으로써"
- ❌ "향상", "극대화", "대폭", "효과적으로", "성공적으로"
- ❌ "완벽히", "혁신적인", "획기적인", "첨단", "안정적인"
- ❌ "및" 과다 사용 (쉼표로 대체)
- ❌ "설계 및 구축", "개선 및 최적화" 같은 나열

#### 좋은 표현 패턴
- ✅ 간결한 명사구: "Redis 캐싱으로 API 응답 속도 50% 개선"
- ✅ 구체적 숫자: "월 100만 건 처리", "응답 시간 0.3초", "비용 90% 절감"
- ✅ 비즈니스 결과: "운영 공수 70% 절감", "장애 대응 시간 50% 단축"
- ✅ 쉼표 구분: "설계, 구현" (대신 "설계 및 구현" X)

#### descriptions 작성 원칙
- 완벽한 문장보다 간결하고 핵심적인 표현
- 기술 스택과 함께 **무엇을 했는지** 명확히
- 불필요한 접속사, 수식어 제거

#### achievements 작성 원칙
- **측정/계산 가능한 수치만 사용** (%, 건수, 시간, 비용)
- 모호한 표현 금지: "유의미하게", "대폭", "크게" 등
- **비즈니스 임팩트 중심**: 기술이 아닌 결과에 집중
- **측정 근거를 주석으로 반드시 명시**: 면접 대비 및 신뢰성 확보

**측정 근거 작성 예시:**
```typescript
achievements: [
  '운영 공수 70% 절감',
  // 측정: 수작업 시간 월 40시간 → 자동화 후 월 12시간 (40-12)/40 = 70%
  'API 응답 시간 1.2초 → 0.3초 개선',
  // 측정: CloudWatch 메트릭 before/after 비교 (7일간 평균)
  '월 API 비용 90% 절감',
  // 측정: AWS 청구서 $1000 → $100 확인
  'DB 쓰기 트래픽 30% 이상 절감',
  // 측정: CloudWatch RDS 메트릭 WriteIOPS before/after 비교
],
```

**측정 방법별 가이드:**
- **실측 데이터**: CloudWatch, APM 툴, 로그, 청구서 등 명시
- **계산 수치**: 계산식 작성 (before/after 값 포함)
- **추정치**: 추정 근거와 방법 설명

#### 예시

**나쁜 예 (AI 티남):**
```
'Redis 캐싱 전략 및 쿼리 튜닝을 통한 대용량 핫딜 데이터 조회 API 성능 개선'
'안정적인 서비스 운영을 위한 크론 서버 헬스체크 및 실시간 장애 감지 파이프라인 구축'
'효율성을 극대화하여 운영 공수를 대폭 절감'
```

**좋은 예 (자연스러움 + HR 관점):**
```
'Redis 캐싱, 쿼리 튜닝으로 API 응답 시간 1.2초 → 0.3초 개선'
'크론 서버 헬스체크, 장애 감지 자동화'
'운영 공수 70% 절감'
```

#### 프리랜서/외주 경력 작성 원칙

**핵심: 간략하게, 기술 스택 중심으로**

프리랜서/외주 경력은 협업, 문제 해결 장기 프로세스를 보여주기 어렵기 때문에 **간략하게** 작성합니다.

**작성 방식:**
- descriptions: 주요 프로젝트를 1줄씩 간단히 나열 (5개 이내)
- achievements: **제거** (외주는 성과보다 기술 경험)
- skillKeywords: 핵심 기술 스택만 나열

**작성 예시:**
```typescript
{
  title: '뉴소프트',
  positions: [{
    title: '프리랜서 풀스택 개발',
    startedAt: '2017-01',
    endedAt: '2023-05',
    descriptions: [
      'Python 기반 부동산 전월세 신고 자동화 서비스 개발',
      'PHP Laravel 기반 가상화폐 선물 거래소 백엔드 개발',
      'Solidity 기반 Polygon 네트워크 DeFi Farm 스마트 컨트랙트 구현',
      'C# 기반 마케팅 자동화 도구 개발',
      'React, Node.js 기반 장기렌트카 견적 비교 플랫폼 개발',
    ],
    skillKeywords: ['PHP', 'Laravel', 'Python', 'C#', 'React', 'Node.js', 'Solidity', 'MySQL', 'Redis'],
  }],
}
```

**왜 간략하게?**
- 최근 회사 경력(팀 협업, 문제 해결)에 집중하기 위함
- 외주는 기술 스택 다양성만 보여주는 보조 역할
- 이력서는 양보다 질 - 핵심 경력을 부각시켜야 함

### 테크 리더가 말하는 매력적인 이력서 (실전 채용 관점)

> 잡플래닛 테크 리더가 연간 100건 이상의 이력서를 검토하며 정리한 실전 채용 인사이트

#### 1. 이력서 구조: 간결한 개요식 작성

**길게 나열 ❌ → 단위별로 구조화 ✅**

각 프로젝트를 다음 항목으로 간결하게 정리:
- **프로젝트명**: 명확하고 간결하게
- **진행기간**: 월 단위까지 정확히 명시
- **개발환경**: 핵심 기술 스택만 (과도한 나열 금지)
- **본인의 역할**: 구체적으로 (예: "백엔드 API 설계, 구현")
- **기여도**: 정량적으로 (예: "팀 3명 중 API 70% 담당")
- **성과**: 측정 가능한 비즈니스 임팩트

**중요**: 지원 포지션과 관계없는 프로젝트는 과감히 제거하거나 뒤로 배치

#### 2. 문제 해결 과정 강조

**잘한 일만 나열 ❌ → 문제 분석과 개선 과정 ✅**

모든 개발은 100% 완벽할 수 없습니다. 오히려 문제 상황에 어떻게 대처했는지가 더 중요합니다.

**작성 예시:**
```typescript
descriptions: [
  '트래픽 증가로 API 응답 시간 3초까지 증가',
  '쿼리 분석 결과 N+1 문제 발견, JOIN으로 개선',
  'Redis 캐싱 적용으로 응답 시간 0.3초로 단축',
],
```

**왜 중요한가**: 면접관은 당신이 문제 상황에 직면했을 때 어떻게 분석하고 해결하는지 알고 싶어합니다.

#### 3. Git과 기술블로그로 신뢰도 높이기

이력서만으로 풀어내기 어려운 부분을 보완:

- **GitHub/GitLab**: 작성한 코드 히스토리 공개
  - 커밋 메시지가 명확하고 일관성 있는지
  - 코드 리뷰 참여 이력
  - 오픈소스 기여 경험

- **기술 블로그**: 학습 과정과 문제 해결 기록
  - 프로젝트에서 겪은 기술적 챌린지와 해결 방법
  - 새로 배운 기술에 대한 정리
  - 꾸준히 업데이트되는지 (성장 욕구 확인)

**면접관이 보는 것**: 같이 일할 때 어떤 스타일일지, 자기계발을 게을리하지 않는지

#### 4. 주체성과 역할 명확히

**"스킬셋 나열" ❌ → "주체적인 기여" ✅**

단순히 "Java, Spring Boot 사용"이 아니라:
- "API 설계 단계부터 참여하여 RESTful 원칙 기반 엔드포인트 정의"
- "팀 내 코드 리뷰 프로세스 제안, 도입"
- "신규 팀원 온보딩 가이드 작성"

**중요**: 특정 언어나 프레임워크 경험보다 **얼마나 주체적으로 프로젝트에 임했는지**가 더 중요합니다.

#### 5. 협업과 리딩 경험 구체화

**모호한 표현 ❌ → 구체적인 역할 ✅**

나쁜 예:
- "팀원들과 협업하여 프로젝트 완수"

좋은 예:
- "프론트엔드 팀과 API 스펙 정의 회의 주도, Swagger 문서 작성"
- "주간 기술 공유 세션 운영 (팀 5명, 매주 목요일)"
- "코드 리뷰어로 참여, 평균 리뷰 소요 시간 24시간 이내 유지"

#### 6. 트래픽과 도메인 경험 강조

**채용 기업과의 핏(Fit) 판단 기준:**

- **같은 도메인 경험**: 이커머스, 금융, 소셜 등
- **비슷한 트래픽 규모 경험**:
  - 대규모 (DAU 100만+): 트래픽 처리, 장애 대응, 모니터링 경험
  - 중소규모/스타트업: 처음부터 끝까지 서비스 개발, 론칭 경험

**작성 예시:**
```typescript
descriptions: [
  '일평균 50만 건 트래픽 처리하는 쇼핑몰 API 개발',
  'CloudWatch 알람 기반 장애 대응 자동화',
],
```

#### 7. 재직기간과 장기 관점의 문제 해결

**1년 미만 재직 시 주의**: 개발은 했지만 문제 해결 과정을 충분히 거치지 않았을 가능성

**면접관이 확인하고 싶은 것:**
- 본인이 개발한 프로덕트의 오류를 스스로 개선해본 경험
- 운영 단계에서 발생한 문제를 해결한 경험
- 사용자 피드백 기반 개선 경험

**회사 경험이 부족하다면**: 개인 프로젝트라도 다음을 포함하여 작성
- 어떤 부분을 리딩했는지
- 기획자/디자이너와 어떻게 협업했는지
- 결과물은 어땠는지
- 사후 조치 (버그 수정, 기능 개선)는 어떻게 했는지

#### 8. 기본기 vs 트렌디한 기술

**주니어 개발자의 경우:**

트렌디한 기술 스택 강조 ❌ → 기본 베이스 탄탄함 ✅

- Java, Python, JavaScript 등 **기본 언어에 대한 깊은 이해**
- 자료구조, 알고리즘 이해도
- 운영 중인 서비스를 제대로 팔로우업할 수 있는 능력

**왜 이게 중요한가**:
- 새로운 툴이나 기술은 기존 팀원들이 도입하는 경우가 많음
- 주니어는 먼저 기존 서비스를 이해하고 유지보수할 수 있어야 함

#### 9. 소프트스킬: 긍정적 마인드와 커뮤니케이션

**면접관이 보는 태도:**

- ✅ "이거 할 수 있을 것 같은데요", "될 것 같아요" - 긍정적 사고
- ✅ "도와주세요, 여기서 막혔어요" - 적극적 커뮤니케이션
- ❌ "이건 안 될 것 같은데요", "어려울 것 같습니다" - 부정적 태도

**이력서에서 소프트스킬 드러내기:**
```typescript
descriptions: [
  '요구사항 불명확 시 기획팀과 적극 소통, 3회 이상 스펙 조율',
  '팀 내 기술 공유 세션 제안, 월 2회 진행',
  '신규 입사자 온보딩 멘토링 (3명)',
],
```



### 디렉토리 구조

- `_posts/`: 블로그 글 (YYYY-MM-DD-title.md 형식)
- `_pages/`: 정적 페이지 (about, categories, tags)
- `_config.yml`: Jekyll 설정 (테마, 플러그인, 기본값)
- `_data/navigation.yml`: 네비게이션 메뉴 설정
- `.github/workflows/pages.yml`: GitHub Actions 빌드/배포
