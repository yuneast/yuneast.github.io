2015년부터 2026년까지 11년간 개발을 해왔다. 프리랜서로 시작해서 코인 선물 거래소, 호주 온라인 카지노 외주 개발을 하다가 인플루언서 플랫폼을 창업했다. 이후 공동주택 관리 서비스, 학원 ERP까지. 그런데 돌이켜보니 묘한 패턴이 있었다. 창업한 회사는 수익성이 낮아서 접었고, 취직한 스타트업들은 서비스 인프라가 안정될 때쯤 자금난으로 월급을 못 주겠다고 했다.

## 반복되는 패턴

### 창업: 인플루언서 플랫폼

직접 창업했다. 블로거 영향력 점수 알고리즘을 만들었다. 크롤링, 검색 순위 분석, Redis 캐싱까지 구축했다. 광고주들 반응도 좋았다.

**기술적으로는 안정화됐다:**
- 블로거 1,000명 분석 가능
- Redis 캐싱으로 비용 절감
- API 응답 속도 0.3초

**그런데:**
- 매출은 나왔지만 너무 적었다
- 일하는 양에 비해 수익이 적었다
- 이대로는 성장이 어렵다고 판단

더 큰 곳에서 경험을 쌓고 싶어서 회사를 정리하고 취직했다.

### 첫 번째 취직: 공동주택 관리 서비스

PHP 레거시를 FastAPI로 재개발했다. 3개월 만에 모든 기능 이전했다. 트래픽도 문제없이 처리했다.

**기술적으로는 성공했다:**
- 20개 단지, 수천 세대 서비스
- API 응답 시간 50% 개선
- 주니어 개발자 온보딩도 순조로움

**그런데:**
- 결제 처리 금액은 컸지만 수수료가 낮았다
- 회사가 실제로 가져가는 돈은 거의 없는 구조
- 6개월 후 자금난

또 나왔다.

### 두 번째 취직: 고소작업차 배차 서비스

배차 시스템 아키텍처 설계, Redis 분산 락으로 동시성 제어, GIS 위치 기반 알림 최적화까지 구현했다.

**기술적으로는 잘 만들었다:**
- 일 배차 30-40건 안정 처리
- 분산 락으로 중복 수락 Race Condition 해결
- 위치 기반 알림으로 불필요한 알림 60% 감소

**그런데:**
- 월급은 잘 나왔다
- 회사도 안정적이었다
- 트래픽이 더 이상 늘 생각을 안 했다

**문제:**
성장이 없었다. 일 배차 30-40건에서 멈춰있었다. 최적화할 건 다 했는데, 트래픽이 늘지 않았다. 기술적으로 배울 게 없었다. 고여가는 것 같았다.

1년 반 다니고 나왔다.

### 세 번째 취직: 학원 ERP

크론 서버 모니터링 자동화, LLM 에러 자동 수정 파이프라인까지 구축했다. 장애 대응 시간 50% 단축했다.

**기술적으로는 잘 돌아갔다:**
- 5-60개 크론잡 안정 운영
- PHP Fatal Error 자동 수정
- 운영 공수 70% 절감
- 월 40억 원 결제 처리

**그런데:**
- 월 40억 처리했지만 회사가 가져가는 건 거의 없었다
- 결제 중개 수수료만으로는 회사 운영이 어려웠다
- 다시 자금난

패턴이 보였다.

## 깨달은 것

### 1. 기술 완벽성에 몰두하지 마라

나는 기술을 완벽하게 만드는 데 집중했다. 인프라 안정화, 성능 최적화, 모니터링 구축. 그런데 회사는 망했다.

- API 응답 속도 0.3초로 만들었지만, 사용자는 1초여도 상관없었다
- 장애 대응 시간 50% 단축했지만, 고객사는 늘지 않았다
- LLM 자동화 구축하는 동안, 영업팀은 신규 고객 확보에 실패했다

**깨달음:**
기술은 비즈니스를 돕는 도구다. 완벽한 기술이 회사를 살리지 않는다. 매출이 회사를 살린다.

### 2. 기술 부채보다 매출이 급하다

내가 기술 부채를 해결하는 동안, 회사는 망해갔다.

**초기 6개월:**
- 기능 개발에 집중
- 버그 투성이지만 일단 작동
- 고객사 확보 집중

**다음 6개월 (내가 한 일):**
- 인프라 안정화에 몰두
- 버그 수정, 모니터링 구축
- 기술 부채 해결에 시간 쏟음

**결과:**
- 인프라는 완벽해짐
- 매출은 그대로
- 회사 자금 바닥

**문제:**
나는 "좋은 코드"를 만들고 있었다. 회사는 "돈 버는 서비스"가 필요했다. 인프라를 다듬는 동안 영업은 멈춰있었고, 회사는 돈이 떨어졌다.

### 3. 월급 지연은 신호다

"이번 달만 참아달라", "투자 유치되면 바로 줄게"

듣기 좋은 말이다. 그런데 한 번 밀리면 계속 밀린다.

**내가 본 패턴:**
- 1개월 지연: "죄송합니다, 다음 달엔 드릴게요"
- 2개월 지연: "투자 유치 직전이에요, 조금만 기다려주세요"
- 3개월 지연: "회사가 어려워졌습니다"

월급이 한 번 밀리면 회사가 회복하는 경우를 못 봤다. 빨리 나오는 게 답이다.

### 4. 안정적이지만 성장 없는 회사도 문제다

고소작업차 배차 서비스에서는 월급도 잘 나왔고, 회사도 안정적이었다. 그런데 나왔다.

**좋았던 것:**
- 월급 지연 없음
- 회사 안정적
- 업무 강도 적당

**그런데:**
- 일 배차 30-40건에서 멈춤
- 트래픽 증가 없음
- 최적화할 건 다 했는데 더 할 게 없음

**깨달음:**
개발자에게 안정만 있고 성장이 없으면 고인다. 트래픽이 늘지 않으면 기술적 챌린지가 없다. 같은 일만 반복하면 배움이 멈춘다.

안정적인 월급도 중요하지만, 성장할 수 있는 환경도 중요하다. 둘 다 없으면 나와야 한다.

### 5. 예외도 있었다: 호주 온라인 카지노

호주에서 워킹홀리데이 비자로 온라인 카지노 개발도 해봤다. 여기는 달랐다.

**기술적으로 재밌었다:**
- 높은 트래픽 처리
- 대용량 데이터 쌓임
- 쿼리 최적화, 성능 개선

**그런데 레거시:**
- ASP로 만들어진 레거시 시스템
- 기술이 너무 낡아서 최적화에 한계

**회사는 안정적이었다:**
- 자본 빵빵
- 규제, 라이센스 문제 없음
- 월급 지연 없음

워킹홀리데이 비자가 끝나서 그만뒀다. 비자만 있었으면 더 다녔을 것 같다.

## 배운 것

### 1. 기술 스택보다 비즈니스 모델을 봐라

예전 면접에서 물어본 것:
- 기술 스택은 뭐 쓰나요?
- 인프라는 어떻게 구성되어 있나요?
- 개발 문화는 어떤가요?
- CI/CD 구축되어 있나요?

**이제 물어볼 것:**
- 현재 매출은? (결제 처리액이 아니라 실제 수익)
- 수익 구조는? (수수료율은 얼마나 되는지)
- 고객사 이탈률은?
- 신규 고객 확보 속도는?
- 런웨이(현금 소진 기간)는?

**왜 중요한가:**
월 40억 결제 처리해도 회사가 망할 수 있다. 수수료가 1%면 40억 처리해도 회사가 가져가는 건 4,000만 원이다. 여기서 인건비, 서버비 빠지면 남는 게 없다.

좋은 기술 스택, 깨끗한 코드, 완벽한 인프라. 이런 건 회사가 돈을 벌 때 의미있다. 회사가 망하면 아무 소용없다.

### 2. 완벽한 코드보다 돌아가는 비즈니스

개발자는 "깨끗한 코드", "좋은 아키텍처"를 원한다. 나도 그랬다. 그런데 회사는 그게 급하지 않았다.

**회사에 돈이 있을 때:**
- 리팩토링 해라
- 인프라 개선해라
- 기술 부채 해결해라

**회사에 돈이 없을 때:**
- 최소한만 고쳐라
- 매출 만드는 것만 해라
- 완벽함은 나중에

나는 회사 상황을 안 보고 기술 완벽성에 집중했다. 회사는 "돈 버는 기능"을 원했는데, 나는 "완벽한 인프라"를 만들고 있었다.

**교훈:**
기술 부채는 회사를 안 망하게 한다. 하지만 매출이 없으면 회사는 망한다. 우선순위를 착각하지 마라.

### 3. 11년 동안 기술은 늘었다

회사는 망했지만 기술은 늘었다.

**2015년:**
- PHP, Python 기본 문법
- MySQL 쿼리 작성
- 프리랜서로 외주 개발

**2026년:**
- FastAPI, Django, Spring Boot 경험
- Redis 캐싱, 쿼리 튜닝
- CI/CD, 모니터링 구축
- LLM 자동화 파이프라인

회사가 망해도 기술은 남는다. 다음 회사에서 쓸 수 있다.

### 4. 스타트업은 도박이다

성공할 수도 있고 망할 수도 있다. 그런데 확률은 망하는 쪽에 가깝다.

**받아들여야 할 것:**
- 월급 지연 가능성
- 회사 폐업 가능성
- 스톡옵션 휴지 조각 가능성

**대비해야 할 것:**
- 비상금 6개월치
- 다른 회사 면접 준비
- 기술 블로그, GitHub 관리

스타트업 다니면서 "이 회사는 성공할 것 같아"라고 생각하지 마라. 언제든 나갈 준비를 해둬라.

## 지금

11년 동안 여러 회사를 다녔다. 성공한 회사는 없었다. 그래도 후회는 없다.

**기술적 성장:**
- 다양한 도메인 경험 (카지노, 인플루언서, 공동주택, 학원)
- 레거시 재개발 경험
- 스타트업 0→1 경험

**더 중요한 깨달음:**
- 기술 완벽성에 몰두하지 마라
- 비즈니스가 안 되면 기술도 소용없다
- 개발자도 매출을 생각해야 한다
- 좋은 코드보다 돈 버는 서비스

**현실적 교훈:**
- 월급 지연은 신호다
- 스타트업은 도박이다
- 회사는 망해도 기술은 남는다

## 마무리

"인프라가 안정될 때쯤 문 닫는 회사들"

패턴이 보였다. 내가 기술을 완벽하게 만들 때쯤, 회사가 망했다. 기술은 좋았는데 비즈니스가 안 됐다.

**개발자의 착각:**
- 좋은 코드를 만들면 회사가 성공한다? 아니다.
- 인프라를 안정화하면 고객이 늘어난다? 아니다.
- 기술 부채를 해결하면 매출이 오른다? 아니다.

**현실:**
매출이 없으면 아무것도 소용없다. 완벽한 기술로 만든 서비스도, 돈을 못 벌면 망한다.

**이제 보는 것:**
- 기술 스택보다 비즈니스 모델
- 인프라 완벽함보다 고객 확보 속도
- 코드 품질보다 수익 구조

11년 차 개발자가 됐다. 기술은 늘었다. 하지만 더 중요한 걸 배웠다. 개발자도 비즈니스를 알아야 한다. 기술만 잘해서는 회사가 안 망한다. 비즈니스가 잘돼야 회사가 안 망한다.

다음 회사는 기술 스택을 먼저 보지 않을 것이다. 매출을 먼저 볼 것이다.
